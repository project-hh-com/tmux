전체 실행 흐름 — 6-Phase 직렬 파이프라인
🔍
Phase A
Discovery — DB 기반 개선 대상 발굴
5종 쿼리를 psql로 직접 실행해 개선이 필요한 키워드를 자동 발굴합니다. rank(검색량)가 낮을수록 우선 선정.
ZERO_RESULT — 결과 0건 ETC_TYPE — 분류 불가 NO_SYNONYM — 동의어 없음 POSSIBLE_FALLBACK — 발골사전 미연결 WRONG_PRIORITY — 우선순위 역전
출력: discovery.json · phase-a-discovery.md
🔬
Phase B
Analysis — 키워드별 구체적 수정 방향 제안
Phase A 발굴 결과를 바탕으로 각 키워드를 DB에서 직접 조회하며 구체적인 수정값을 결정합니다. 단순 진단이 아닌 실행 가능한 수치 제안.
target_keyword_id 탐색 brand_id / product_midtype_ids 조회 proposed_priority 계산
출력: analysis.json (각 키워드별 fix_type + 구체 수정값) · phase-b-analysis.md
🗣️
Phase C
Plan Debate — 수정 방향 도메인 적합성 토론
Red(반론) → Blue(방어/대안) → Judge(판정) 구조로 각 제안의 도메인 적합성을 검증합니다. SQL 안전성이 아닌 검색 의미 정확성에 집중.
🔴 Red — 타입 오류·동의어 의미 차이·발골 ID 불일치 공격 🔵 Blue — DB 증거 기반 방어 또는 수정 대안 제시 ⚖️ Judge — APPROVE / MODIFY / REJECT 판정
출력: debate_plan.json (approved_proposals + rejected_proposals) · phase-c-debate.md
⚡
Phase D
Propose — 승인된 플랜 기반 SQL 생성 (5 에이전트 병렬)
Phase C에서 APPROVE/MODIFY된 항목만 처리. 5개 에이전트가 diagnosis_code를 기준으로 자신의 담당 항목만 필터링해 병렬 실행합니다. 해당 항목이 없으면 빈 SQL + 완료 신호 자동 생성.
1️⃣ D1 zero_fixer — ZERO_RESULT 2️⃣ D2 type_fixer — ETC_TYPE 3️⃣ D3 synonym_fixer — NO_SYNONYM 4️⃣ D4 anchor_fixer — POSSIBLE_FALLBACK 5️⃣ D5 priority_fixer — WRONG_PRIORITY
출력: d1_zero_fixer.sql · d2_type_fixer.sql · d3_synonym_fixer.sql · d4_anchor_fixer.sql · d5_priority_fixer.sql
📦
Phase E
Ship-Ready — 최종 SQL 패키징 + 매니페스트
D1~D5 SQL을 하나의 트랜잭션으로 통합. 적용 순서 보장: UPDATE_PRIORITY → UPDATE_KEYWORD_TYPE → INSERT_SYNONYM → UPDATE_ANCHOR. 토큰 매칭 기준이 먼저 확정되어야 이후 변경이 일관성 유지.
출력: ship.sql (단일 BEGIN/COMMIT) · ship-manifest.md (변경 요약 + 롤백 SQL)
📊
Phase F
Validate — 예상 개선도 추정 + 최종 리포트
ship.sql 적용 시 기대 효과를 fix_type별 논리에 따라 추정합니다. SIMILAR 동의어 추가 시 Tier2 폴백 노출 예상, POSSIBLE_FALLBACK 해소 시 정밀 필터 전환 효과 등.
출력: phase-f-validate.md (키워드별 현재/예상 결과수 · 적용 권장도)

🔎 진단 코드 — 개선 분류 체계
ZERO_RESULT
D1 zero_fixer
사전에 등록은 됐지만 결과 0건. 주로 동의어 관계 부재가 원인. 처방: 결과 있는 상위 키워드와 SIMILAR 관계 추가 → Tier2 폴백으로 노출.
INSERT
SIMILAR
ETC_TYPE
D2 type_fixer
keyword_type이 ETC라 발골 파이프라인이 동작 안 함. 처방: 올바른 타입(BRAND/CATEGORY/COLOR)으로 변경 + category_main 수정 + 발골사전 연결.
UPDATE
TYPE
NO_SYNONYM
D3 synonym_fixer
SAME 동의어가 없어 Tier1 검색 범위가 자기 자신뿐. 처방: 동의어 키워드와 SAME 관계 추가 → Tier1 확장. Debate에서 SIMILAR가 더 적합하면 변경.
INSERT
SAME
POSSIBLE_FALLBACK
D4 anchor_fixer
BRAND/CATEGORY/COLOR인데 발골사전 미연결 → 텍스트 검색만 사용. 처방: brand_id 또는 product_midtype_ids 연결 → 정밀 필터 시나리오로 전환.
UPDATE
ANCHOR
WRONG_PRIORITY
D5 priority_fixer
토큰 매칭 순서 이상. priority(낮을수록 먼저)가 1순위로 작동.
케이스 1: rank 상위인데 priority=100(기본값) → 중요 키워드가 후순위 매칭.
케이스 2: "자라"(BRAND)와 "자라체크"(CATEGORY)가 동일 priority → "자라체크" 쿼리에서 "자라"가 먼저 매칭 → 브랜드 시나리오로 잘못 처리.
UPDATE
PRIORITY
Tier1 / Tier2 폴백 구조
Tier 1 — SAME 동의어
사용자 쿼리와 매칭된 키워드의 SAME 관계 키워드까지 합산. 항상 실행.
예: "청바지" → "데님팬츠"(SAME) 상품도 함께 노출
Tier 2 — SIMILAR 폴백
Tier1 결과가 0건일 때만 실행. SIMILAR 관계 키워드로 확장.
예: "린넨셔츠" Tier1 0건 → "셔츠"(SIMILAR) 상품으로 폴백

skq가 조회하는 주요 테이블
search_keywords 키워드 사전 · priority · keyword_type
search_keyword_relations 동의어·유사어 관계 (SAME/SIMILAR)
search_keyword_rankings result_product_count · rank · score
search_brand_keywords 브랜드 발골사전 (brand_id)
search_category_keywords 카테고리 발골사전 (product_midtype_ids)
search_color_keywords 색상 발골사전 (variant_ids)
사용법
# 최초 1회 — skq 환경 셋업 (sq-setup과 DB 자격증명 공유) bash ~/development/tmux/skq/skq-setup # 기본 실행 — 전체 검색 키워드 품질 개선 skq # 범위 지정 실행 skq "ZERO_RESULT 키워드 집중 개선 — 0건 키워드 top 20" skq "rank 50위 이내 키워드 priority 최적화" skq "브랜드 키워드 발골사전 미연결 일괄 해소" # SQL 결과 확인 후 적용 ship --list # 적용 가능한 skq 결과 목록 ship --dry-run # 최신 결과 SQL 미리보기 (DB 미적용) ship # 최신 결과 적용 ('ship' 입력 재확인) ship skq-20260504-153000 # 특정 실행 결과 적용
핵심 특징
  • 6-Phase 직렬 파이프라인 — A→B→C→D→E→F 순으로 실행. Phase D만 5개 에이전트가 병렬 실행.
  • tmux 패널 모드 — 실행 중인 에이전트가 오른쪽 패널에 실시간으로 표시. Phase 완료 시 패널 자동 정리.
  • 비 tmux 환경 지원 — 백그라운드 세션 모드로 자동 전환. done 플래그 폴링 방식은 동일.
  • 도메인 집중 토론 — Phase C Debate는 SQL 안전성이 아닌 검색 의미 정확성(타입·동의어 관계·발골 ID)만 검증.
  • 자율 필터링 — 각 D-Phase 에이전트가 자신의 diagnosis_code를 스스로 필터링. 해당 없으면 빈 SQL + 자동 완료.
  • SQL 적용 순서 보장 — Phase E에서 UPDATE_PRIORITY → UPDATE_TYPE → INSERT_SYNONYM → UPDATE_ANCHOR 순으로 통합. 토큰 매칭 기준 먼저 확정.
  • 읽기/쓰기 계정 분리 — skq 에이전트는 read-only로만 조회. ship 명령어만 쓰기 계정으로 실제 적용.
  • 수동 적용 확인 — ship 실행 시 'ship' 재입력으로 의도 재확인. 실패 시 트랜잭션 자동 롤백.
  • 롤백 SQL 포함 — ship-manifest.md에 각 변경사항 롤백 SQL 자동 포함.
  • sq와 자격증명 공유 — ~/.config/sq/db.env를 그대로 사용. skq-setup이 sq-setup 설정을 재활용.
