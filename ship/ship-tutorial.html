🚢 ship 설치 가이드

기능 설명 한 줄로 설계·구현·검증·PR까지 자동화하는 Claude Agent Teams 워크플로 설치 튜토리얼

macOS Claude Code CLI tmux gh CLI
ship "기능 설명" ← 한 줄 입력만 하면 됩니다 ↓ Phase 1 Architect 계획 수립 + 에이전트 선택 (plan.md / agents.txt / commit_msg.txt) ↓ Debate 1 Red/Blue/Judge 설계 계획 검토 (최대 2라운드) ↓ Phase 3 Wave A (구현) backend · frontend · debugger ← tmux 병렬 실행 Wave B (검증) reviewer · test · security ← Wave A 완료 후 ↓ Debate 2 Red/Blue/Judge 구현 결과 검토 (REWORK 시 Wave 재실행) ↓ Phase 4 CI 게이트 tsc → eslint → test → build ↓ Phase 5 자동 수정 루프 실패 시 최대 3회 재시도 ↓ Phase 6 Release Manager 커밋 · 푸시 · gh pr create --draft --label "Claude Agent" ↓ CI 통과 후 gh pr ready ← Ready for Review 전환 (수동)

필수 도구

도구용도확인 명령어

ℹ️
ship-setup이 자동 처리
Claude Code CLI, tmux, gh CLI가 미설치여도 bash ship-setup이 자동으로 설치를 안내합니다. 수동으로 미리 설치할 필요는 없습니다.

terminal

2
레포지토리 클론

ship 스크립트는 ~/development/tmux 레포에 포함되어 있습니다.

terminal
# 처음 클론 git clone https://github.com/project-hh-com/tmux.git ~/development/tmux # 이미 있다면 최신 업데이트만 cd ~/development/tmux && git pull
💡
심볼릭 링크의 장점
~/.local/bin/ship이 ~/development/tmux/ship/ship을 직접 가리키므로 git pull만 해도 바로 최신 버전으로 업데이트됩니다.
3
ship-setup 실행

최초 1회만 실행합니다. Claude Code CLI·tmux·gh CLI 설치 확인, GitHub 인증, 에이전트 설정 파일 생성, PATH 등록을 8단계로 처리합니다.

terminal
bash ~/development/tmux/ship/ship-setup
  • 1
    Claude Code CLI 설치 확인 — 미설치 시 npm install -g @anthropic-ai/claude-code를 자동 실행합니다. npm이 없으면 Node.js 설치 안내 후 종료합니다.
  • 2
    Claude Code 인증 확인 — 인증이 필요하면 claude 명령어를 직접 실행해 로그인하세요.
  • 3
    tmux 설치 확인 — 미설치 시 brew install tmux를 자동 실행합니다. tmux가 없으면 에이전트들이 백그라운드 세션으로 전환되어 실행됩니다.
  • 4
    gh CLI 설치 확인 — 미설치 시 brew install gh를 자동 실행합니다.
  • 5
    GitHub 인증 확인 — 미인증 시 지금 인증할지 묻습니다. Release Manager가 PR 생성 시 필요합니다.
  • 6
    ~/.claude/settings.json 생성 — dangerouslySkipPermissions: true를 설정합니다. 에이전트가 매번 권한 확인 없이 파일을 읽고 쓸 수 있게 됩니다.
  • 7
    에이전트 정의 파일 생성 — ~/.claude/agents/에 backend·frontend·test·debugger·reviewer·security 역할 파일을 생성합니다.
  • 8
    PATH 등록 — ~/.local/bin/ship 심볼릭 링크를 생성하고 ~/.zshrc에 PATH를 추가합니다.

실행 단계별 설명

Phase 1 — Architect

입력된 기능 설명을 분석해 세 가지 파일을 생성합니다.

파일내용
.agent/ship-TIMESTAMP/plan.md구현 계획 — 무엇을, 왜, 어떻게, 위험요소와 방어책
.agent/ship-TIMESTAMP/agents.txt필요한 에이전트 목록과 각자 맡을 역할
.agent/ship-TIMESTAMP/commit_msg.txt커밋·PR 제목용 한 줄 요약 (60자 이내)

Debate 1 / Debate 2 — Red/Blue/Judge

Debate 1은 plan.md를 검토하고, Debate 2는 Wave 구현 결과를 검토합니다. Debate 2에서 Judge가 REWORK를 판정하면 Wave 에이전트를 처음부터 재실행합니다.

Phase 3 — Wave A + Wave B

Wave A(구현 에이전트)가 먼저 실행되고, 완료 후 Wave B(검증 에이전트)가 실행됩니다. tmux 환경에서는 각 에이전트가 별도 패널에서 병렬로 실행됩니다.

Phase 4 — CI 게이트

tsc --noEmit → eslint --max-warnings 0 → npm test → npm run build 순서로 실행됩니다. 실패 시 Phase 5 자동 수정 에이전트가 최대 3회 수정을 시도합니다.

Phase 6 — Release Manager

모든 에이전트 리포트를 통합해 커밋·푸시하고 Draft PR을 생성합니다. PR에는 --label "Claude Agent"가 자동으로 붙고, CI 실패 상태면 --draft가 추가됩니다.

💡
PR 생성 후
CI 게이트가 통과하면 gh pr ready로 Draft → Ready for Review로 전환하세요. prd/master를 베이스로 한 PR은 Release Manager가 자동으로 차단합니다.
7
에이전트 선택 기준

Architect가 기능 설명을 분석해 필요한 에이전트만 자동으로 선택합니다.

요청 유형선택 에이전트 (Wave)
버그 / 이슈 수정debugger (A) — 반드시 포함
API / 서비스 로직 변경backend (A)
UI 컴포넌트 변경frontend (A)
새 코드 추가test (B)
코드 품질 / 리팩토링reviewer (B)
인증 / 인가 / 시크릿 관련security (B)
ℹ️
불필요한 에이전트는 Architect가 자동으로 제외합니다. 예를 들어 간단한 버그 수정이면 debugger + backend만 선택하고 frontend나 security는 포함하지 않습니다.
8
tmux 레이아웃

tmux 세션 안에서 ship을 실행하면 에이전트들이 나란히 배치되어 실시간 진행 상황을 확인할 수 있습니다.

tmux 패널 레이아웃
┌──────────────────┬──────────────────┬──────────────────┐ │ │ 🔧 backend │ 👀 reviewer │ │ [Plan/Main] │ (Wave A) │ (Wave B) │ │ ├──────────────────│──────────────────│ │ ship 진행 상황 │ 🐛 debugger │ 🧪 test │ │ 출력 중 │ (Wave A) │ (Wave B) │ │ │ │ │ └──────────────────┴──────────────────┴──────────────────┘

Wave A 에이전트들이 먼저 왼쪽 컬럼에 표시되고, Wave A 완료 후 Wave B가 오른쪽 컬럼에 추가됩니다. Wave 완료 시 패널이 자동으로 정리됩니다.

💡
tmux 없이도 실행 가능
tmux 세션 밖에서 실행하면 에이전트들이 백그라운드 tmux 세션으로 자동 생성됩니다. tmux attach -t ship-backend 등으로 각 에이전트 세션에 접속해 진행 상황을 확인할 수 있습니다.
