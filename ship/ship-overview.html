ship "기능 설명" 한 줄이면 Architect가 계획을 세우고, Wave A/B 에이전트들이 병렬로 구현·검증하며, CI 게이트를 통과하면 Release Manager가 Draft PR을 자동 생성합니다.

ship 실행 플로우
🏗️
Phase 1
Architect — 설계 & 에이전트 선택
입력된 기능 설명을 분석해 plan.md(구현 계획), agents.txt(필요 에이전트 목록), commit_msg.txt(커밋 메시지)를 생성합니다. 불필요한 에이전트는 자동으로 제외합니다.
plan.md agents.txt commit_msg.txt
🗣️
Debate 1
설계 계획 토론 — Red / Blue / Judge
Red 에이전트가 plan.md의 허점과 위험요소를 공격합니다. Blue 에이전트가 각 항목을 ACCEPT/MITIGATE/REJECT로 판정하고 plan.md를 직접 수정합니다. Judge가 CONSENSUS/CONTINUE를 판정해 최대 2라운드로 진행합니다.
🔴 Red 공격 🔵 Blue 방어 ⚖️ Judge 판정
⚡
Phase 3
Wave A (구현) + Wave B (검증) 병렬 실행
tmux 창 안에서 Wave A 에이전트(backend·frontend·debugger)가 먼저 실행됩니다. Wave A 완료 후 Wave B 에이전트(reviewer·test·security)가 구현 결과를 검토합니다. 각 에이전트는 별도 tmux 패널에서 병렬로 실행됩니다.
Wave A: 구현 Wave B: 검증 tmux 병렬 실행
🗣️
Debate 2
구현 결과 토론 — REWORK 판정 허용
Wave 결과를 Red/Blue/Judge가 검토합니다. Judge가 REWORK를 판정하면 Wave 에이전트를 처음부터 재실행합니다(최대 REWORK 횟수 제한). CONSENSUS 판정 시 다음 단계로 진행합니다.
REWORK → Wave 재실행 CONSENSUS → 다음 단계
🔍
Phase 4
하네스 CI 게이트 — 4단계 검증
tsc → eslint → test → build 순서로 4단계 CI 게이트를 실행합니다. 실패 시 Phase 5 자동 수정 에이전트가 최대 3회 수정을 시도합니다.
tsc --noEmit eslint --max-warnings 0 npm test npm run build
🚀
Phase 6
Release Manager — 커밋·푸시·PR 자동 생성
모든 에이전트 리포트를 통합해 최종 커밋·푸시를 수행하고 GitHub Draft PR을 생성합니다. PR 본문에는 요청 내용, 설계 계획 요약, 에이전트 리포트, CI 게이트 상태가 포함됩니다. --label "Claude Agent"가 자동으로 붙습니다.
gh pr create --draft --label "Claude Agent"
에이전트 목록
🔧 backend Wave A
API Route Handler, 서비스 레이어, Zod 검증 구현. services/→types/ 단방향 import 준수.
🎨 frontend Wave A
React 컴포넌트(PascalCase), TanStack Query, Tailwind CSS 구현.
🐛 debugger Wave A
경쟁 가설 방식으로 버그 원인 분석 후 수정. 버그/이슈 요청 시 반드시 포함.
👀 reviewer Wave B
가독성, 타입 안전성, 레이어 규칙 위반, 중복 코드 검토.
🧪 test Wave B
Vitest + Testing Library 단위/통합 테스트. 커버리지 80% 이상 필수.
🔒 security Wave B
OWASP Top 10, 인증/인가, 시크릿 노출, SQL 인젝션 검토.

Architect가 요청 내용을 분석해 필요한 에이전트만 선택합니다. 버그 수정이면 debugger, UI 변경이면 frontend, 새 코드 추가면 test가 포함되는 방식입니다.

Debate 패턴 — Red / Blue / Judge
🗣️ Debate 1 — 설계 검토
🔴
Red: plan.md의 허점, 위험요소, 누락된 엣지케이스를 공격합니다. 해결책 제시 없이 문제만 지적.
🔵
Blue: 각 공격을 ACCEPT/MITIGATE/REJECT로 판정하고 plan.md를 직접 수정.
⚖️
Judge: CONSENSUS(종료) 또는 CONTINUE(다음 라운드) 판정. 최대 2라운드.
🗣️ Debate 2 — 구현 검토
🔴
Red: Wave A/B 결과물의 문제점과 누락 사항을 공격.
🔵
Blue: 공격을 검토하고 코드를 직접 수정.
⚖️
Judge: REWORK 판정 가능 — Wave 에이전트를 처음부터 재실행. CONSENSUS면 다음 단계로.
CI 하네스 게이트 (Phase 4)
1
타입 검사 (TypeScript)
npx tsc --noEmit
2
린트 (ESLint)
npx eslint . --max-warnings 0
3
테스트 (Vitest / Jest)
npm test -- --passWithNoTests
4
빌드 검증
npm run build

게이트 실패 시 Phase 5 자동 수정 에이전트가 로그를 분석하고 코드를 수정한 뒤 게이트를 재실행합니다. 최대 3회 반복 후에도 실패하면 Draft 상태로 PR을 생성합니다.

PR 자동 생성 (Phase 6 — Release Manager)

## 요청
<ship에 입력한 기능 설명>

## Summary
<plan.md 기반 2-3줄 요약>

## 에이전트 리포트
<각 에이전트 산출물 핵심 요약>

## CI 게이트
상태: 통과 / 실패(Draft)

🤖 Generated with Claude Agent Ship
--label "Claude Agent" --draft (기본값) base: main (또는 migration/tailwind)

CI 통과 후 gh pr ready로 Draft → Ready for Review로 전환합니다. prd/master 베이스 PR 생성은 금지됩니다.
