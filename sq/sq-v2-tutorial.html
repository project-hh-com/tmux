📊 sq-v2 튜토리얼

사전 준비부터 결과 해석까지 — sq-v2로 검색 품질을 처음 측정하는 분을 위한 단계별 가이드

품질 측정 3-Wave diagnosis_codes 500 키워드
Wave A runner ← 키워드 500개 수집 (웹서칭 250 + DB rank 상위 250) + 검색 API 배치 호출 ↓ Wave B scorer ← v2 채점 체계로 결과 채점 → scores.json (diagnosis_codes / query_type 포함) ↓ Wave C reporter ← 집계 → report.md / detail.md / detail.csv 생성 ↓ 📋 결과 해석 diagnosis_codes로 개선 방향 파악 → sq-v3로 수정 실행

실행 흐름 이해

sq-v2는 3개의 Wave를 순차적으로 실행합니다. 각 Wave는 .done 파일로 완료를 표시합니다.

Wave A 🏃 runner — 키워드 수집 & API 호출 (약 5~10분)

웹서칭으로 트렌드 키워드 250개, DB에서 rank 상위 키워드 250개를 수집하고 검색 API를 배치 호출합니다.

키워드 수집 전략
# 웹서칭 250개 — 최신 트렌드/시즌 키워드 # DB 250개 — rank 상위 (서비스 대표 키워드) # # → 검색 API 배치 호출 (최대 500건) # → results/{keyword}.json 저장
💡
v1 대비: SQ_SEARCH_PARAMS에서 불필요한 step=0&search_source=POPULAR_KEYWORD 파라미터 제거. API 스키마에 존재하지 않는 파라미터였습니다.
Wave B 📈 scorer ★ v2 개편 — 채점 (약 15~25분)

results/*.json을 v2 채점 체계로 채점합니다. 6개 항목 100점 만점, diagnosis_codes 배열, query_type 유형 분류 포함.

scores.json — 항목별 설명
{ "keyword": "린넨셔츠", "score": 42, "grade": "D", "segment": "ZERO_RESULT", "query_type": "E", // A/B/C/D/E "keyword_type": "ETC", "diagnosis_codes": ["ZERO_UNRECOGNIZED", "ETC_TYPE"], "diagnosis": "키워드 사전 미등록...", "breakdown": { "결과존재성": 0, "검색범위적절성": 0, "의도파악정확도": 0, "결과관련성": 0, "상위노출품질": 0, "품절노출": 5 } }
Wave C 📝 reporter ★ v2 개편 — 리포트 생성 (약 5~10분)

scores.json을 집계해 리포트 3종을 생성합니다. 백엔드 알고리즘 컨텍스트와 diagnosis_codes → 수정 대상 테이블 매핑이 추가되었습니다.

생성 파일
docs/reports/sq2-{TIMESTAMP}/ ├── search-quality-report.md # 요약 리포트 (백엔드 컨텍스트 포함) ├── search-quality-detail.md # 키워드별 상세 (diagnosis_codes 포함) └── search-quality-detail.csv # 분석용 CSV

report.md 구조

search-quality-report.md 섹션 구성
§1 실행 개요 (일시, 브랜치, 목적) §2 전체 품질 점수 요약 (평균 점수, 등급 분포) §3 ZERO_RESULT 분석 (UNRECOGNIZED vs NO_MATCH) §4 ETC_TYPE 키워드 목록 §5 POSSIBLE_FALLBACK 목록 §6 우선 개선 대상 (F등급, D등급 상위) §7 백엔드 알고리즘 컨텍스트 ← v2 신규 §8 diagnosis_codes → 수정 대상 매핑 ← v2 신규

scores.json 직접 분석

terminal — jq 활용
FLAG_DIR=.agent/sq2-{TIMESTAMP} # 등급별 분포 jq 'group_by(.grade) | map({grade: .[0].grade, count: length})' "${FLAG_DIR}/scores.json" # diagnosis_codes별 키워드 목록 jq '[.[] | select(.diagnosis_codes | contains(["ZERO_UNRECOGNIZED"])) | .keyword]' \ "${FLAG_DIR}/scores.json" # F등급 키워드만 jq '[.[] | select(.grade == "F") | {keyword, score, diagnosis_codes}]' \ "${FLAG_DIR}/scores.json" # POSSIBLE_FALLBACK 건수 jq '[.[] | select(.segment == "POSSIBLE_FALLBACK")] | length' \ "${FLAG_DIR}/scores.json"

점수 해석 방법

diagnosis_codes → 개선 방향 매핑

diagnosis_code의미개선 방향
ZERO_UNRECOGNIZED 키워드가 search_keywords 사전에 미등록 sq-v3: INSERT_KEYWORD
ZERO_NO_MATCH 발골 성공했으나 결과 0건 — 매핑 문제 sq-v3: ADD_SYNONYM / FIX 계열
ETC_TYPE keyword_type = ETC — must 필터 미적용 sq-v3: FIX_KEYWORD_TYPE
NO_SYNONYM SAME 관계 0건 — Tier 1 결과 단조로움 sq-v3: ADD_SYNONYM
POSSIBLE_FALLBACK Tier 2 fallback 의심 (extracted 없음) 조사 후 INSERT_KEYWORD 또는 방치
BRAND_MAPPING_WRONG 브랜드 발골 성공 but 상품 브랜드 불일치 sq-v3: FIX_BRAND_MAPPING
LOW_SIMILAR SIMILAR 관계 0건 — Tier 2 다양성 부족 sq-v3: ADD_SIMILAR

평균 점수 해석 시 주의

ℹ️
ZERO_RESULT는 평균에서 제외됩니다
평균 점수는 GENERAL + POSSIBLE_FALLBACK 기준입니다. ZERO_RESULT 비율이 높을수록 실제 품질은 평균보다 나쁠 수 있습니다. report.md §3에서 ZERO_RESULT 비율을 반드시 확인하세요.

v1 리포트와 직접 비교 불가

⚠️
점수 체계가 다릅니다
v1과 v2는 채점 항목과 가중치가 다르므로 scores.json을 직접 비교할 수 없습니다. sq-v2 최초 실행 후 이 시점을 baseline으로 확립하세요.

건강한 점수 분포 예시

지표양호 수준우선 점검 필요
전체 평균 점수70점 이상60점 미만
ZERO_RESULT 비율10% 미만20% 이상
ETC_TYPE 비율5% 미만15% 이상
POSSIBLE_FALLBACK 비율15% 미만30% 이상
F등급 비율5% 미만15% 이상

자주 묻는 질문

특정 Wave가 실패했을 때 해당 Wave부터 재실행할 수 있나요?

terminal
FLAG_DIR=.agent/sq2-{TIMESTAMP} # scorer(Wave B)부터 재실행 rm "${FLAG_DIR}/scorer.done" # reporter(Wave C)만 재실행 rm "${FLAG_DIR}/reporter.done" # sq-v2 재실행 sq-v2

sq와 sq-v2를 동시에 실행해도 되나요?

네. sq는 코드 수정(opensearch/api 에이전트), sq-v2는 품질 측정(runner+scorer+reporter)이라 역할이 다릅니다. 단, 같은 브랜치에서 동시에 실행하면 커밋 충돌이 날 수 있으므로 별도 브랜치를 권장합니다.

POSSIBLE_FALLBACK이 너무 많아요. 정상인가요?

POSSIBLE_FALLBACK은 v1의 FALLBACK과 다르게 F등급 강제가 없습니다. 채점은 계속됩니다. 다만 비율이 30% 이상이면 search_keywords 사전이 충분하지 않다는 신호입니다. sq-v3로 INSERT_KEYWORD를 실행하면 비율을 낮출 수 있습니다.

reporter만 다시 실행하고 싶으면?

terminal
FLAG_DIR=.agent/sq2-{TIMESTAMP} \ SQ_OUTPUT_DIR=docs/reports/sq2-{TIMESTAMP} \ reporter-v2

sq-v2 결과를 sq-v3 실행에 어떻게 활용하나요?

sq-v2의 scores.json에서 diagnosis_codes를 확인하세요. ZERO_UNRECOGNIZED가 많으면 sq-v3에서 INSERT_KEYWORD 중심으로, NO_SYNONYM이 많으면 ADD_SYNONYM 중심으로 목적을 지정해 실행하면 효과적입니다.

고급 사용법

sq-v2 → sq-v3 연계 워크플로

terminal — 권장 사용 패턴
# 1단계: sq-v2로 현황 측정 sq-v2 "현재 검색 품질 baseline 측정" # 2단계: 결과 확인 후 개선 방향 결정 cat docs/reports/sq2-{TIMESTAMP}/search-quality-report.md # 3단계: sq-v3로 데이터 수정 sq-v3 "ZERO_UNRECOGNIZED 키워드 INSERT 및 NO_SYNONYM 키워드 동의어 추가" # 4단계: SQL 검토 후 적용 # (fix_preview.md 확인 → write 계정으로 실행) # 5단계: sq-v2로 재측정 (개선도 확인) sq-v2 "sq-v3 적용 후 품질 재측정"

특정 diagnosis_code 키워드만 필터링

terminal
FLAG_DIR=.agent/sq2-{TIMESTAMP} # ZERO_UNRECOGNIZED 키워드 목록 추출 jq '[.[] | select(.diagnosis_codes | contains(["ZERO_UNRECOGNIZED"])) | .keyword]' \ "${FLAG_DIR}/scores.json" # ETC_TYPE이고 rank 있는 키워드 (수정 효과 높음) jq '[.[] | select(.diagnosis_codes | contains(["ETC_TYPE"])) | select(.rank != null)]' \ "${FLAG_DIR}/scores.json" # query_type별 평균 점수 jq 'group_by(.query_type) | map({type: .[0].query_type, avg: (map(.score) | add / length)})' \ "${FLAG_DIR}/scores.json"

두 시점 비교 (개선도 측정)

terminal
# before: sq-v3 적용 전 jq '[.[] | .score] | add / length' .agent/sq2-20260501-before/scores.json # after: sq-v3 적용 후 jq '[.[] | .score] | add / length' .agent/sq2-20260504-after/scores.json # 키워드별 점수 변화 (공통 키워드 비교) jq -s '[.[0][] as $b | .[1][] | select(.keyword == $b.keyword) | {keyword: .keyword, before: $b.score, after: .score, delta: (.score - $b.score)}] | sort_by(-.delta)' \ .agent/sq2-before/scores.json \ .agent/sq2-after/scores.json
💡
브랜치 이름으로 before/after 관리
sq-v2 실행 시 브랜치 이름에 before / after를 포함하면 PR 리뷰에서 개선도를 한눈에 확인할 수 있습니다. (예: sq2/after-keyword-fix-20260504)
