Wave A: opensearch · api · runner Wave B: scorer · reporter

CONSENSUS → 다음 단계 CONTINUE → 다음 라운드 REWORK → Wave 재실행

⚠️
이중 의도 쿼리 (Dual Intent Query) 주의
"진스"처럼 카테고리(진스 바지)이면서 동시에 브랜드명(타미진스)을 포함하는 검색어의 경우, 두 의도가 모두 결과에 반영되어야 정상입니다.
한쪽 의도만 노출되면 이중 의도 커버리지 점수 0점 처리됩니다.
판별 기준: search_category_keywords + search_brand_keywords 양쪽에 동일 keyword_id가 존재하는 경우.
🚨
폴백(전체 반환) 우선 감지
danble 검색 API는 매칭 실패 시 전체 상품을 반환(폴백)합니다. result_count가 커도 실제 검색 실패.
판정 기준: total_count > 3000 → 즉시 F등급 · FALLBACK segment로 처리, 이후 채점 건너뜀.
🔎
결과 존재 여부
검색 결과 1건 이상 (폴백 제외) → 20점 / 0건 → 0점
20점
✅
관련성 ★ 핵심
상위 min(결과수, 10)개 상품 중 키워드와 관련된 상품 비율 × 30점 (연속값)
관련 판정: 상품명·브랜드명·카테고리명에 키워드 포함 여부 (텍스트 매칭 + AI 판단)
예: 10개 중 7개 관련 → 7/10 × 30 = 21점
30점
🎯
이중 의도 커버리지
DB 조회 필수 — search_category_keywords + search_brand_keywords 양쪽 확인 후 점수 부여
카테고리 의도 (10점) 해당 카테고리 상품 결과에 포함 여부
브랜드 의도 (10점) 해당 브랜드 상품 결과에 포함 여부
단일 의도(카테고리 XOR 브랜드만) → 20점 만점 처리 | 미등록 키워드 → 0점
20점
📍
상위 노출 적절성
가장 관련성 높은 상품이 1~3위 → 15점 / 4~7위 → 8점 / 8~10위 → 4점 / 11위 밖 → 0점
15점
📦
품절 상품 노출
product_displays.is_sold_out 기준. 품절률 0% → 15점 / 1~10% → 12점 / 11~30% → 6점 / 31% 이상 → 0점
15점
리포트 세그먼트 분리
GENERAL — 정상 결과 (평균 점수 산출 기준) FALLBACK — 폴백 반환 (별도 집계) ZERO_RESULT — 결과 0건 (별도 집계)
등급 기준 A 85~100 B 70~84 C 50~69 D 30~49 F 0~29
Debate 시스템 — Red / Blue / Judge
🔴
Red (공격자)
계획/분석의 허점, 위험요소, 누락된 엣지케이스를 공격. 해결책 제시 금지 — 문제 지적만.
🔵
Blue (방어자)
각 공격 항목을 ACCEPT / MITIGATE / REJECT로 판정. ACCEPT·MITIGATE는 실제 파일을 수정해 반영.
⚖️
Judge (심판)
Red/Blue 토론을 심판. Debate 2에서는 REWORK 판정으로 Wave 전체 재실행 가능 (횟수 제한 없음).

주요 검색 관련 테이블
search_keywords 키워드 마스터 · rank · score
search_logs 사용자 검색 로그 · 최근 패턴

product_display_keywords 상품-키워드 매핑
product_displays 상품 전시 정보 · sale_status
search_keyword_relations 연관 키워드 · relation_score
SELECT only · INSERT/UPDATE/DELETE 불가
# 최초 1회 — DB 환경 셋업 (psql 설치 + 연결 테스트 + PATH 등록) bash ~/development/tmux/sq/sq-setup # 이후 sq 실행 시 ~/.config/sq/db.env 자동 로드 sq "검색 품질 개선 내용" # DB 직접 접속 (확인용) source ~/.config/sq/db.env psql -h "${PGHOST}" -U "${PGUSER}" -d "${PGDATABASE}" --no-password

# tmux 세션에서 실행 (패널 분할을 위해 필수) tmux new-session -s work # 기본 실행 — 인자 없이 sq만 입력하면 기본 평가 자동 실행 sq ← "40~50대 남성 패션 키워드 500개 검색 품질 평가" 자동 적용 # 범위 지정 실행 — 특정 카테고리·목적에 집중 sq "정장/수트 키워드 집중 평가 — 이전 리포트에서 F등급이 18개였던 카테고리" sq "OpenSearch 검색 쿼리 가중치 조정으로 카테고리 매칭 정확도 향상" sq "검색 결과 0건 비율 개선 — step 파라미터 및 limit 수정"
핵심 특징
  • 인자 없이 sq만 입력하면 "40~50대 남성 패션 키워드 500개 검색 품질 평가" 자동 실행 — 범위 지정 시 sq "..."로 덮어쓰기 가능
  • 리포트는 docs/reports/sq-{timestamp}/에 저장 — 실행한 현재 폴더 기준 (예: ~/development/danble/search-quality/docs/reports/sq-20260429-172913/). 해당 폴더의 git 레포에 함께 커밋/푸시됨
  • 요청 내용에 따라 Architect가 필요한 에이전트만 자동 선택
  • 브랜치 생성 → 중간 커밋 → PR 생성이 모든 실행에서 자동으로 보장
  • Wave A(분석/실행) → Wave B(평가/보고) 순차 실행 — 데이터 수집 완료 후 점수·리포트 생성
  • Debate 2에서 REWORK 판정 시 Wave를 횟수 제한 없이 재실행
  • 게이트 실패해도 Draft PR 자동 생성 — 팀에 진행 상황 공유
  • 모든 프롬프트는 파일 경로 참조 방식 — OS 인자 한도 초과 없음
  • 에이전트가 danble 운영 DB에 직접 쿼리 — 실제 키워드·점수·상품 데이터 기반 분석
  • DB 자격증명은 ~/.config/sq/db.env에만 보관 — git 저장소에 절대 포함되지 않음
