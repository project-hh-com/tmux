Agent 파일 설치

sq가 선택해서 실행하는 에이전트 정의 파일입니다. Wave A(분석/실행) / B(평가/보고)로 분류됩니다.

에이전트Wave역할
🔎 opensearchWave AOpenSearch 쿼리 DSL, boost/weight 최적화, 인덱스 매핑 검토
🔌 apiWave A검색 API 파라미터 분석, 스크립트 수정, 응답 파싱 검증
🏃 runnerWave A자동 검색 실행, 50개 단위 배치, JSON 결과 즉시 저장
📊 scorerWave B100점 만점 품질 점수 산정, A/B/C/D/F 등급 배분. 평가 항목: 결과존재(20)·관련성 연속값(30)·이중의도 DB필수(20)·상위노출(15)·품절노출(15). 폴백(total_count > 3000) 우선 감지 → F/FALLBACK 처리. LLM이 각 상품 관련성을 직접 Y/N 판단 (문자열 매칭 코드 금지). relevance_ratio 항상 0.0~1.0 소수점 비율로 계산 (0.0 고정 금지). 결과를 Write 툴로 scores.json에 직접 저장. 절대 금지: score.py·batch_score.sh 등 스크립트 생성, psql을 Python 경유 실행
📝 reporterWave Bsearch-quality-report.md · search-quality-detail.csv → docs/reports/sq-{timestamp}/ 저장 (실행 폴더 기준). scores.json이 유일한 채점 소스 — 집계·포매팅만 수행. GENERAL/FALLBACK/ZERO_RESULT 세그먼트 분리 통계 포함. 절대 금지: 독립 채점, scorer_reporter.py 등 독립 스크립트 생성, 점수 수정

sq 스크립트 설치
terminal
# 1. 레포 클론 (처음 한 번, ship과 공유) git clone https://github.com/project-hh-com/tmux.git ~/development/tmux # 2. 심볼릭 링크 생성 — 소스 파일 하나만 관리 mkdir -p ~/.local/bin ln -sf ~/development/tmux/sq/sq ~/.local/bin/sq # 3. 실행 권한 chmod +x ~/.local/bin/sq # 4. PATH 설정 (ship과 동일하게 이미 설정되어 있으면 스킵) echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc source ~/.zshrc which sq # → /Users/[이름]/.local/bin/sq
💡
심볼릭 링크의 장점
~/.local/bin/sq은 ~/development/tmux/sq/sq을 직접 가리켜요. 소스 파일 하나만 관리하면 되고, git pull만 해도 바로 최신 버전으로 실행됩니다.
🔄
업데이트 방법
cd ~/development/tmux && git pull — 끝. ship과 sq 모두 한 번에 업데이트됩니다.

Git 워크플로 동작 방식

sq는 실행 시작부터 끝까지 git 작업을 자동으로 처리합니다.

  • 1
    브랜치 자동 생성 — main·master·prd·develop에서 실행하면 sq/{slug}-{YYYYMMDD-HHmmss} 브랜치를 자동 생성. 이미 feature 브랜치에 있으면 현재 브랜치 사용.
  • 2
    중간 커밋/푸시 — Wave A 완료 후 (sq: ... — 분석), Wave B 완료 후 (sq: ... — 평가), 자동 수정 라운드마다 자동 커밋. 변경사항 없으면 스킵.
  • 3
    PR 생성 (항상 실행) — CI 게이트 통과 여부와 무관하게 Phase 6에서 항상 PR을 생성. 게이트 실패 시 Draft PR, 통과 시 일반 PR. 라벨 Search Quality 자동 부착.
  • 4
    릴리스 로그 최종 커밋 — 릴리스 매니저 종료 후 SUMMARY.md와 TIMELINE.md가 확정되면 git add -A → commit → push 자동 실행. 팀원들이 PR에서 전체 타임라인과 실행 요약을 확인할 수 있음.
🚫
PR 베이스 브랜치 보호
릴리스 매니저는 prd·master를 베이스로 한 PR을 절대 생성하지 않습니다. 해당 상황이 감지되면 즉시 중단하고 사유를 보고합니다.
🗄️
DB 환경 설정 — sq-setup

sq 에이전트들은 danble PostgreSQL read-only replica에 직접 쿼리해 실제 키워드·점수·상품 데이터를 기반으로 분석합니다. 자격증명은 git에 포함되지 않고 로컬 파일에만 보관됩니다.

🔐
보안 구조
자격증명은 ~/.config/sq/db.env (chmod 600)에만 저장됩니다. git 저장소의 sq, sq-setup 파일에는 비밀번호가 없으며 .gitignore로 env 파일 유출을 차단합니다.

sq-setup 실행

terminal
# 최초 1회만 실행 (이후 sq 실행 시 자동으로 db.env 로드) bash ~/development/tmux/sq/sq-setup

실행하면 순서대로 진행됩니다:

  • 1
    DB 자격증명 입력 — Host / Database / User / Password(비표시) / Port를 대화식으로 입력. Enter로 기본값 사용. 입력한 값이 ~/.config/sq/db.env에 chmod 600으로 저장됨.
  • 2
    psql 설치 확인 — 미설치 시 brew install libpq 자동 실행. PATH도 자동 추가.
  • 3
    DB 연결 테스트 — search_keywords 카운트 쿼리로 실제 접속 확인. 주요 검색 테이블 6개의 row 수를 출력.
  • 4
    sq PATH 등록 — ~/.local/bin/sq 심볼릭 링크 생성. PATH에 없으면 ~/.zshrc 자동 추가.
🔗
에이전트 DB 자동 연결
sq가 각 에이전트를 실행할 때 생성하는 .sh 스크립트에 source ~/.config/sq/db.env가 자동 포함됩니다. tmux 새 패널은 부모 환경을 상속하지 못해 PGHOST 등이 비어있을 수 있는데, 이 방식으로 모든 에이전트가 DB에 자동 접속합니다. sq-setup 최초 1회만 실행하면 이후 모든 에이전트 실행 시 자동 연결됩니다.

에이전트별 DB 활용

에이전트DB 활용 방식
🔎 opensearch search_keywords의 auto_score/md_score 분포 조회 · result_product_count = 0 키워드 분석
🔌 api search_logs에서 최근 검색 패턴 분석 · search_keyword_rankings rank 분포 확인
🏃 runner search_keywords에서 rank 순 키워드 리스트 직접 추출 · 최근 7일 인기 검색어 보완
📊 scorer search_keywords로 keyword_id 조회 · search_category/brand_keywords로 이중의도 판별 · product_displays로 품절률(soldout_rate) 측정. LLM이 각 상품 관련성 직접 Y/N 판단 후 Write 툴로 scores.json 저장.
📝 reporter search_keyword_relations 커버리지 · DB 기준 기대 상품수 vs 실제 결과수 비교 · 세그먼트별(GENERAL/FALLBACK/ZERO_RESULT) 분리 집계 포함. scores.json만 참조하여 집계·포매팅 수행 (독립 채점 금지).
⚠️
VPN / 네트워크
RDS read replica는 회사 네트워크 또는 VPN에서만 접근 가능할 수 있습니다. DB 연결 실패 시 VPN 연결을 먼저 확인하세요. 에이전트는 DB 없이도 동작하지만 분석 품질이 저하됩니다.

eval-only 실행 (스냅샷 비교)

opensearch/api 에이전트 없이 runner+scorer+reporter만 실행하면 eval-only 모드로 동작합니다. Wave A에 DB 변경 에이전트가 없으므로 before/after DB 스냅샷 대신 이전 run의 scores.json과 비교합니다.

terminal — eval-only 실행 예시
# 평가만 실행 (OpenSearch/API 변경 없이 점수·리포트만) sq "40~50대 남성 패션 키워드 500개 검색 품질 평가" # → runner + scorer + reporter만 선택됨 (eval-only) # → Wave B 완료 후 이전 run scores.json과 자동 비교 # → 개선/하락 키워드 top 10 출력
📊
eval-only 스냅샷 비교 출력 항목
이전 run과 현재 run을 자동으로 비교해 다음을 출력합니다: 평균 점수 변화 · 세그먼트별 분포(GENERAL/FALLBACK/ZERO_RESULT) · 등급 분포(A~F) · 개선 키워드 top 10 · 하락 키워드 top 10. 이전 run 데이터가 없으면 현재 run만 출력합니다.
