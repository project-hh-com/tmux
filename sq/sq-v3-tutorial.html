실행 흐름 이해

sq-v3는 4개의 Wave를 순차적으로 실행합니다. 각 Wave가 완료(`.done` 파일 생성)되어야 다음 Wave가 시작됩니다.

Wave A 🏃 runner — 키워드 수집 & API 호출 (약 5~10분)

DB에서 4가지 유형의 키워드를 PHASE 1 쿼리로 수집하고 검색 API를 배치 호출합니다.

PHASE 1 쿼리 — 키워드 수집 전략
# R-1: ZERO_RESULT 150개 → runner_zero_result.csv # R-2: ETC_TYPE 100개 → runner_etc_type.csv # R-3: NO_SYNONYM 100개 → runner_no_synonym.csv # R-4: POSSIBLE_FB 100개 → runner_possible_fallback.csv # + 웹서칭 트렌드 50개 # → 검색 API 배치 호출 (최대 500건) # → results/*.json 저장
Wave B 📊 scorer — 채점 & fixable 필드 생성 (약 10~20분)

results/*.json을 채점하고 v3 신규 필드(fixable/fix_type/fix_priority)가 포함된 scores.json을 생성합니다.

scores.json — v3 추가 필드 (각 키워드 항목)
{ "keyword": "린넨셔츠", "segment": "ZERO_RESULT", "diagnosis_codes": ["ZERO_UNRECOGNIZED"], // --- v3 신규 필드 --- "fixable": true, "fix_type": "INSERT_KEYWORD", "fix_priority": "HIGH" }
fix_priority조건
HIGHZERO_RESULT + rank 있음 (수정 기대효과 명확)
MEDIUMETC_TYPE / NO_SYNONYM / POSSIBLE_FALLBACK
LOWrank 없음 / ADD_SIMILAR만 필요
NONENOT_FIXABLE — SQL 생성 대상 아님
Wave C 🔧 keyword_fixer ★ 신규 — SQL 생성 (약 10~30분)

scores.json의 fixable=true 키워드마다 PHASE 2 컨텍스트 쿼리(F-1 ~ F-6)를 실행하고 SQL 파일을 생성합니다.

⚠️
현재 read-only 단계
SQL 파일만 생성합니다. DB에 직접 실행하지 않습니다. 생성된 SQL은 .agent/sq3-*/fix_search_keywords.sql에 저장됩니다.
PHASE 2 컨텍스트 쿼리 (키워드당 실행)
# F-1: 키워드 전체 컨텍스트 (brand/category/relation 매핑) # F-2: 브랜드명 → brand_id 조회 # F-3: product_type 목록 조회 # F-4: 유사 키워드 존재 확인 # F-5: relation_score 범위 확인 # F-6: 같은 category_main 키워드 탐색 # # → fix_search_keywords.sql 생성 (BEGIN/COMMIT 포함) # → fix_preview.md 생성
fix_type대상 테이블설명
INSERT_KEYWORDsearch_keywords미등록 키워드 신규 삽입
ADD_SYNONYMsearch_keyword_relationsSAME 관계 추가 (Tier 1 다양화)
ADD_SIMILARsearch_keyword_relationsSIMILAR 관계 추가 (Tier 2 보강)
FIX_KEYWORD_TYPEsearch_keywordskeyword_type ETC→CATEGORY/BRAND 수정
FIX_BRAND_MAPPINGsearch_brand_keywords브랜드 매핑 오류 수정
FIX_CATEGORY_MAPPINGsearch_category_keywords카테고리 매핑 오류 수정
NOT_FIXABLE—자동 수정 불가 (SQL 미생성)
Wave D 📝 reporter — 최종 리포트 생성 (약 5~10분)

reporter-v3를 사용해 최종 리포트 3종을 생성합니다. scores.json이 유일한 채점 소스이며 집계·포매팅만 수행합니다.

생성 파일
docs/reports/sq3-{TIMESTAMP}/ ├── search-quality-report.md # §7 수정계획 + §8 SQL 미리보기 포함 ├── search-quality-detail.md # fix_type / fix_priority 컬럼 추가 └── search-quality-detail.csv # fix_type / fix_priority 컬럼 추가

fix_preview.md 읽는 법

keyword_fixer가 생성한 수정 계획 요약입니다. SQL 적용 전에 반드시 이 파일로 내용을 확인하세요.

fix_preview.md — 형식
| fix_type | keyword | 변경 내용 | 근거 | 예상 효과 | |----------------|----------|--------------------------------------|-----------------------------|--------------------| | INSERT_KEYWORD | 린넨셔츠 | search_keywords INSERT (CATEGORY) | ZERO_UNRECOGNIZED, rank 42 | 검색 결과 생김 | | ADD_SYNONYM | 맨투맨 | SAME 관계 추가 → target: 스웨트셔츠 | NO_SYNONYM, SAME 0건 | Tier 1 결과 다양화 | | FIX_KEYWORD_TYPE| 에코백 | keyword_type ETC→CATEGORY | ETC_TYPE, rank 17 | must 필터 활성화 |

SQL 구조 이해

fix_search_keywords.sql — 구조
-- ============================================================ -- fix_search_keywords.sql -- 생성일: 2026-05-04 | sq-v3 | read-only 생성 → 검토 후 적용 -- ⚠️ read-only 계정으로는 실행 불가. write 권한 계정으로 실행하세요. -- ============================================================ BEGIN; -- [INSERT_KEYWORD] 린넨셔츠 INSERT INTO search_keywords (keyword, normalized_keyword, keyword_type, category_main) VALUES ('린넨셔츠', '린넨셔츠', 'CATEGORY', '상의'); -- [ADD_SYNONYM] 맨투맨 ↔ 스웨트셔츠 INSERT INTO search_keyword_relations (source_keyword_id, target_keyword_id, relation_type, relation_score) VALUES (123, 456, 'SAME', 0.9); -- [FIX_KEYWORD_TYPE] 에코백: ETC → CATEGORY UPDATE search_keywords SET keyword_type = 'CATEGORY' WHERE id = 789; COMMIT;

scores.json 직접 분석

terminal — jq 활용

SQL 검토 & 적용

sq-v3는 현재 read-only 단계입니다. SQL 파일을 직접 검토한 후 write 계정으로 실행해야 합니다.

  • 1
    fix_preview.md 검토 — 수정 내용이 의도와 맞는지 각 fix_type별로 확인. 의심스러운 항목은 SQL 파일에서 주석 처리.
  • 2
    SQL 파일 검토 — INSERT/UPDATE 대상 ID가 올바른지 확인. 불필요한 항목은 삭제하거나 --로 주석 처리.
  • 3
    write 계정으로 실행 — 검토 완료 후 write 권한 계정으로 psql에서 SQL 실행. BEGIN/COMMIT 트랜잭션으로 보호됨.

수동 실행 방법

terminal — write 계정 필요
# ⚠️ write 권한 계정으로 실행 (read_only 계정 아님) FLAG_DIR=.agent/sq3-{TIMESTAMP} # SQL 미리 확인 cat "${FLAG_DIR}/fix_search_keywords.sql" # 실행 (write 계정) PGPASSWORD="..." psql -h "${PGHOST}" -U danble_write -d danble -p "${PGPORT}" \ -f "${FLAG_DIR}/fix_search_keywords.sql"
🚫
주의
SQL에는 BEGIN/COMMIT이 포함되어 있습니다. 실행 전 반드시 내용을 확인하세요. ROLLBACK이 필요한 경우를 대비해 snapshot_before_fix.csv를 백업으로 보관하세요.

검토 체크리스트

  • ☐
    fix_preview.md에서 각 키워드의 변경 내용이 올바른지 확인
  • ☐
    INSERT_KEYWORD: keyword_type이 BRAND/CATEGORY/ETC 중 올바른 타입인지 확인
  • ☐
    ADD_SYNONYM: source/target 키워드 관계가 실제로 동의어인지 확인
  • ☐
    FIX_BRAND_MAPPING: brand_id가 실제 브랜드와 일치하는지 brands 테이블에서 확인
  • ☐
    SQL COMMIT 전 EXPLAIN이나 SELECT로 영향 범위 재확인
  • ☐
    snapshot_before_fix.csv 보관 (롤백 참고용)

자주 묻는 질문

특정 Wave가 실패했을 때 해당 Wave부터 재실행할 수 있나요?

.done 플래그 파일을 삭제하면 해당 Wave부터 재실행됩니다.

terminal
FLAG_DIR=.agent/sq3-{TIMESTAMP} # keyword_fixer(Wave C)부터 재실행 rm "${FLAG_DIR}/keyword_fixer.done" # scorer(Wave B)부터 재실행 rm "${FLAG_DIR}/scorer.done" # sq-v3 재실행 sq-v3

keyword_fixer가 생성한 SQL에 오류가 있으면?

fix_search_keywords.sql을 직접 편집해 문제 있는 항목을 삭제하거나 주석 처리 후 실행하세요. sq-v3를 다시 실행할 필요 없이 SQL 파일만 수정하면 됩니다.

sq-v2와 같이 실행해도 되나요?

네. sq-v2는 품질 측정, sq-v3는 데이터 수정을 담당하므로 역할이 다릅니다. sq-v2로 전체 품질 현황을 파악한 후 sq-v3로 수정하는 순서를 권장합니다.

REWORK가 발동되는 조건은?

조건동작
게이트 실패 (scores.json 없음, SQL 파일 없음 등)해당 Wave 에이전트 재실행 (MAX_REWORK=3)
토론(Red/Blue/Judge) 결과 REWORK 판정전체 Wave 처음부터 재시작
REWORK_COUNT ≥ 3강제 종료

reporter만 다시 실행하고 싶으면?

terminal
FLAG_DIR=.agent/sq3-{TIMESTAMP} \ SQ_OUTPUT_DIR=docs/reports/sq3-{TIMESTAMP} \ reporter-v3

keyword_fixer가 너무 오래 걸려요

키워드 수가 많을수록 PHASE 2 쿼리도 늘어납니다. scores.json에서 fixable=true인 키워드 중 fix_priority가 HIGH/MEDIUM인 것만 처리하도록 sq-v3 파일의 MAX_FIX_KEYWORDS 값을 줄일 수 있습니다.
