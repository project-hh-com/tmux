검색 품질을 측정·진단하는 평가 워크플로 — v1 채점 오류를 수정하고 백엔드 실제 로직을 정확히 반영

버전 비교
sq (v1)
코드 수정
OpenSearch DSL·API 코드를 직접 수정하는 개발 워크플로. 코드 변경 + PR 생성.
sq-v2
품질 측정
검색 품질을 측정·진단하는 평가 워크플로. 백엔드 로직을 정확히 반영한 채점 체계.
sq-v3
데이터 수정
search_keywords 관련 테이블을 수정하는 개선 워크플로. SQL 파일 생성 + 검토 후 적용.
💡
권장 사용 순서
sq-v2로 품질 현황 측정 → sq-v3로 데이터 수정 → sq-v2로 재측정. sq-v2는 측정 도구이고, sq-v3는 수정 도구입니다.

4
v1 대비 주요 변경
항목sq v1 (기존)sq-v2 (수정)이유
FALLBACK 처리 total_count > 1000 → F등급 강제 POSSIBLE_FALLBACK 표시 후 채점 계속 백엔드 Tier 2는 result_count=0일 때만 발동. total_count>1000 기준은 코드 어디에도 없음
브랜드+카테고리 충돌 이중 의도로 양쪽 모두 채점 카테고리 우선 처리 (유형 B) token_classifier.py가 충돌 시 카테고리 우선, 브랜드 제거
브랜드 관련성 extracted_brands 있으면 자동 관련성 1.0 LLM이 각 상품 독립 Y/N 판단 발골 성공 ≠ 관련성 만점. 브랜드 필터 내 품질 차이가 숨겨짐
ZERO_RESULT 단일 ZERO_RESULT로 통합 ZERO_UNRECOGNIZED / ZERO_NO_MATCH 구분 수정 방향이 완전히 다름. 미등록 키워드 vs 매핑 문제
진단 필드 diagnosis 단일 문자열 diagnosis_codes 배열 + diagnosis 서술 복합 진단 가능, sq-v3의 fix_type 매핑 기반
ETC 타입 처리 단일 의도로 20점 채점 0점 + ETC_TYPE 진단 코드 키워드 사전에 없는 키워드는 검색 품질 측정 불가
API 파라미터 step=0&search_source=POPULAR_KEYWORD 포함 page=1&limit=50만 사용 존재하지 않는 파라미터 제거
5
진단 코드 (diagnosis_codes)

scores.json 각 항목에 배열로 포함됩니다. sq-v3의 fix_type 결정에도 직접 사용됩니다.

ZERO_UNRECOGNIZED
extracted_* 모두 비어있음. 키워드가 search_keywords 사전에 미등록.
→ sq-v3: INSERT_KEYWORD
ZERO_NO_MATCH
extracted_* 있었지만 결과 0건. 엔터티 발골 성공했으나 매칭 상품 없음.
→ sq-v3: ADD_SYNONYM / FIX_BRAND_MAPPING
ETC_TYPE
keyword_type이 ETC. 카테고리나 브랜드로 분류되지 않아 must 필터 미적용.
→ sq-v3: FIX_KEYWORD_TYPE
NO_SYNONYM
SAME 관계 0건. Tier 1 결과가 단일 키워드에만 의존.
→ sq-v3: ADD_SYNONYM
POSSIBLE_FALLBACK
extracted 없음 + total_count > 2000. Tier 2 fallback 의심 (v1의 FALLBACK과 다름 — 채점 계속).
→ sq-v3: INSERT_KEYWORD 또는 조사 필요
BRAND_MAPPING_WRONG
브랜드로 발골되었으나 상품이 해당 브랜드 소속이 아님. 브랜드 매핑 오류 의심.
→ sq-v3: FIX_BRAND_MAPPING
CATEGORY_MAPPING_WRONG
카테고리로 발골되었으나 결과 상품의 카테고리 불일치.
→ sq-v3: FIX_CATEGORY_MAPPING
LOW_SIMILAR
SIMILAR 관계 0건 또는 부족. Tier 2 결과 다양성 부족.
→ sq-v3: ADD_SIMILAR

게이트 체크
✅
검색 결과 JSON ≥ 100개 — results/ 디렉토리에 100개 이상 JSON 파일 존재
✅
scores.json 존재 — Wave B scorer 정상 완료
✅
리포트 파일 3개 존재 — report.md / detail.md / detail.csv
ℹ️
F등급 0건 비율 — 정보성 게이트. F등급이 많아도 실패 처리하지 않음 (현황 파악용)
🔄
REWORK 조건 — 게이트 실패 시 해당 Wave 재실행 (MAX_REWORK=3). 토론 REWORK 판정 시 전체 Wave 재시작
